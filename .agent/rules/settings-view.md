# Settings View

## Module Structure

```
views/settings/
├── mod.rs              — State, Message, Action, update; `SettingsTab` enum (6 variants), `route_write` helper
├── entries.rs          — Entry building / filtering / cross-tab search
├── items.rs            — Re-exports `SettingValue` / `SettingMeta` / `SettingItem` / `SettingsEntry` from the data crate, plus the `MacroRows::{new, take, finish}` helper used by the per-tab builders. Every builder ends with `macro_rows.finish()` (debug-panics naming any un-taken key; warns in release); conditional rows follow the take-unconditionally-push-conditionally convention so consumption is data-independent. Construct `SettingMeta` via `SettingMeta::new(key, label, category).with_subtitle(...)`
├── items_general.rs    — Library, Display, Behavior, Window & Tray, Advanced, Account
├── items_interface.rs  — Navigation, Slot List, Player Bar, Font & Icons (`icon_set` set picker), Metadata Strip, Artwork Overlays, Artwork Column
├── items_playback.rs   — Transitions, Volume Normalization, Scrobbling, Rating Reminder, Playlists
├── items_hotkeys.rs    — Per-category hotkey entries
├── items_theme.rs      — Mode, Display, and the Select-Theme picker opener only. Theme COLORS are edited directly in each theme's TOML file (`~/.config/nokkvi/themes/`), not in the GUI — `build_theme_items` emits a single `__theme_picker` row (`ActivateKind::ThemePicker`) whose value is the current theme name
├── items_visualizer.rs — Frame, Signal, Bars, Lines, Scope rows all come from the Visualizer `define_settings!` table via `MacroRows`; headers, the Restore sentinel, and the Bar Colors Dark/Light `ColorArray` sections stay hand-written
├── sub_lists.rs        — Three sub-list states: `SubListState` (color gradient editor — in-place detail-pane replacement) + `FontSubListState` (font picker — modal overlay) + `ThemeSubListState` (theme picker — modal overlay; each `ThemeRow` carries pre-resolved `ThemePreviewColors` so the render path never does disk IO, selecting via `SettingsAction::ApplyPreset { stem, display_name }`)
├── presets.rs          — Theme discovery + apply: `all_theme_files()` (→ `Vec<(ThemeInfo, ThemeFile)>` via `theme_loader::discover_theme_files`, for the picker swatches), `apply_theme(stem)` (writes `theme = "stem"` to config.toml), `active_theme_stem()`
├── rendering.rs        — Slot rendering: detail-row chrome, headers, items, colors, hotkey badges, toggle sets, plus `render_theme_slot` (paints each theme picker row in that theme's OWN palette via `ThemeRow.preview`)
├── sentinel.rs         — `SentinelKind` enum for stringly-typed pseudo-keys. Now only `{Logout, RestoreVisualizer, RestoreAllHotkeys}`. Activation intercepts these via `SentinelKind::from_key` and returns dedicated `SettingsAction`s — they never reach the `route_write` value-edit path
├── test_support.rs     — `#[cfg(test)]`-only helpers (`extract_keys`, `header_labels`, `assert_section_keys` are `pub(crate)`; `section_slice` is a private helper they share) used by the items/items_theme test modules. Structure tests pin section headers + per-section ordered KEY lists (failures name the diverging key), keeping at most one coarse item-count backstop per tab
└── view.rs             — Persistent two-pane layout: 340 px categories sidebar + scrollable detail pane (or narrow-variant horizontal chip strip below `NARROW_BREAKPOINT_WIDTH = 1400 px`). Hosts the modal pickers: `render_font_*` and `render_theme_modal` both go through the shared `render_picker_modal` scaffold
```

Settings dispatch tables for **General / Interface / Playback / Visualizer** are generated by `define_settings!` (`data/src/services/settings_tables/`) — the macro emits the dispatch arm, the apply/dump/write round-trips, and (per `ui_meta` entry) the UI row, so adding a settable knob means adding one entry there rather than touching the legacy match arm. The consolidated `columns.rs` invocation owns all 50 view-column copy-steps (compile-time exhaustive destructure), and `copy_only` entries own the non-dispatchable residuals (published in `TAB_*_COPY_ONLY_KEYS`). Theme / Hotkey items and the visualizer color sections still live in their items_*.rs files.

## Architecture

- 6 tabs in display order: **General, Interface, Playback, Hotkeys, Theme, Visualizer**
- Persistent sidebar drives `active_category: SettingsTab`; the detail pane renders `cached_entries` **verbatim** — `SettingsPage::view(window_width, window_height)` takes only the two window dims and never rebuilds entries. The update path owns all rebuilds via `refresh_entries` (`build_category_sections` / `search_all_entries`) and `Nokkvi::refresh_settings_entries_if_dirty` (gated on `current_view == Settings`, fires when `config_dirty || cached_entries.is_empty()`). Rebuild triggers: view entry (`handle_switch_view`), tab switch, search edits, every `SettingsAction` write (tail of `handle_settings`), theme/visualizer/settings hot-reloads, hotkey rebind, the chrome light-mode + crossfade toggles, the `SetLightModeAtomic` side-effect arm in `dispatch_settings_side_effect` (must rebuild AFTER the atomic flips — `handle_player_settings_loaded` runs earlier in the same dispatch and rebuilds against the old mode), the reset-visualizer dialog, the Local Music Path dialog commit (`update/text_input_dialog.rs`), the default-playlist picker, and `handle_player_settings_loaded`. Two writers deliberately mark dirty WITHOUT refreshing because they cannot co-occur with an open Settings view: the artwork-vertical-height drag (`handle_artwork_vertical_drag`, runs in a library view) and the Playlists-view "Set as Default Playlist" action. **Any new writer of settings-visible state must set `settings_page.config_dirty = true` and (if it can run while Settings is open) call `refresh_settings_entries_if_dirty()`** — there is no per-frame fallback to mask a missed mark.
- Sidebar nav: `SidebarUp` / `SidebarDown` (Shift+Backspace / Shift+Tab by default) move the sidebar cursor; `SidebarClickItem(idx)` jumps. The sidebar is a fixed six-row column rather than a slot list — six rows always fit, so `sidebar_slot_list.viewport_offset` is reused only as the cursor index.
- Narrow variant: when `window_width < NARROW_BREAKPOINT_WIDTH` (1400 px) the layout collapses to a horizontal chip strip + search header above the detail pane.
- Search: lives in the sidebar header. Cross-tab search (`search_all_entries` in `entries.rs`) is fuzzy and relevance-ranked, not a substring filter: each item is scored across label / curated synonyms (`utils::setting_keywords`) / subtitle / category via the `utils::fuzzy` scorer, with clamped tier constants (`LABEL_BASE` 1000 down to `TAB_CONTEXT_BASE` 80; `BINDING_BASE` lets Hotkeys rows match by key combo) so a strong low-tier match never outranks a higher tier. Sections rank by their best item's score; a tab-name match pulls the whole tab in only when `fuzzy::is_strong_score` passes, and ranks below precise hits. Match highlighting is computed at render time from the live query — `SettingItem` carries no transient search state.
- Sub-lists (three): the color-gradient editor replaces the detail pane in place via `SubListState`; the font picker and theme picker each stack as a modal (`FontSubListState` / `ThemeSubListState`), both rendered through the shared `render_picker_modal` scaffold in `view.rs`.

## SettingValue Types

| Type | Interaction |
|------|-------------|
| `Float` / `Int` | ←/→ adjust with clickable arrow buttons |
| `Bool` | Toggle via clickable On/Off badges |
| `Enum` | Cycle via clickable option badges (`EditSetValue`) |
| `Text` | Read-only display row; activating routes structurally through `item.on_activate: Option<ActivateKind>` — `ActivateKind::FontPicker` opens the `FontSubListState` modal (`font_family`), `ThemePicker` opens the `ThemeSubListState` modal (`__theme_picker`), `TextInputDialog` returns `SettingsAction::OpenTextInput` (`general.local_music_path`), `PlaylistPicker` opens the default-playlist picker; `on_activate == None` stays display-only (e.g. server URL) |
| `ToggleSet` | Multi-select badges. ←/→ moves cursor, Enter toggles, ↑/↓ sets on/off |
| `HexColor` | Direct hex input. `HEX_EDITOR_INPUT_ID` + `FocusHexInput` action for auto-focus |
| `ColorArray` | Opens a sub-slot-list for gradient editing |
| `Hotkey` | Badge display + key capture mode (Esc cancels, Delete resets, steal-on-conflict) |

## Key Patterns

- **Typed config write routing** (`src/config_writer.rs`):
  - `ConfigKey::AppScalar` → `config.toml`
  - `ConfigKey::Theme` / `ThemeArrayEntry` → active theme file
  - The dispatch handler matches on the variant — never sniff key prefixes
  - Routes through `route_write` (helper at the top of `views/settings/mod.rs`) which short-circuits `general.*` keys to `WriteGeneralSetting` and forks the remainder into `WriteConfig { ConfigKey::Theme | AppScalar, ... }`; sentinel pseudo-keys (`__action_logout`, `__restore_visualizer`, `__restore_all_hotkeys`) use `SentinelKind` (in `views/settings/sentinel.rs`) instead of stringly keys
- **`verbose_config` mode** (`VerboseConfig` enum: `On` / `Off` / `Clean`, an `Enum` setting; legacy bool configs load via `deserialize_verbose_config_with_bool_compat` — `true` → On, `false` → Off): `SettingsSideEffect::WriteVerboseConfig { mode }` in `update/settings.rs` runs three steps. The macro setter persists to redb first (`set_verbose_config` → `save_redb_only`); the side-effect arm then does a **synchronous** TOML write keyed on the mode — `On` expands via `config_writer::write_full_visualizer`, `Off` strips via `config_writer::strip_to_sparse(false)` (keeping the auto-added `# description` comments), `Clean` strips via `strip_to_sparse(true)` (also removing the `[visualizer]` comments), each with its own success/warn toast — and finally spawns the deferred `write_all_toml_after_verbose_toggle` task for `write_all_toml_public`. The section writers consume `is_verbose_config()` = `verbose_config.writes_all_defaults()` (true only for `On`). Going forward, `handle_settings_write_config` suppresses the surgical `# description` comment whenever `verbose_config.writes_comments()` is false (i.e. `Clean`)
- **Strip visibility toggles** (Metadata Strip section): `ToggleSetToggle` flips the cached entry then emits `WriteGeneralSetting`. The mini-player column (`TrackInfoDisplay::MiniPlayer`) paints its own colored title/artist/album text widgets; the per-segment color/kind plumbing the old progress-bar overlay used has been removed.
- **Artwork Column** (Interface tab): mode (`Auto` / `AlwaysNative` / `AlwaysStretched` / `AlwaysVerticalNative` / `AlwaysVerticalStretched` / `Never`) + stretch fit + draggable width pct. Width drag persists via `WriteGeneralSetting`
- **System Tray** (General tab): `show_tray_icon` + `close_to_tray`. Live-toggle starts/stops the StatusNotifierItem subscription without restart
- **Font picker**: modal overlay (`FontSubListState`, stacked above the detail pane via `render_picker_modal`). System fonts via `font-kit`, `LazyLock`-cached. Search box uses `SettingsMessage::SubListSearchChanged` (shared with the theme picker) + `FONT_SEARCH_INPUT_ID`
- **Theme picker**: modal swatch picker (`ThemeSubListState`, opened by the `__theme_picker` `Browse Themes…` row in `build_theme_items` via `ActivateKind::ThemePicker`). `ThemeSubListState::new` reads + parses every theme file ONCE at open to pre-resolve each row's `ThemePreviewColors`, so each row is painted in its OWN palette (`render_theme_slot` in `rendering.rs`). Rendered through `render_theme_modal` → `render_picker_modal` in `view.rs`, with a `SlotListConfig::without_hover_wash()` config (the accent wash would muddy own-palette rows). Selecting a row emits `SettingsAction::ApplyPreset { stem, display_name }`, keyed by stem (no positional index to drift through the search filter); the root handler applies via `presets::apply_theme` (rewrites `theme = "stem"` in config.toml → hot-reload). The active theme row gets a "● active" marker. Theme COLORS are NOT editable in the GUI — they live in each theme's TOML file
- **Search pitfall**: `SlotListDown` must navigate within `cached_entries` — only `SearchChanged` rebuilds entries
- **Icons**: `SettingItem` and `SettingsEntry::Header` carry an icon path. SVGs in `assets/icons/` are auto-registered by `build.rs`; unknown paths fall back to `play.svg` with a warn log
