# Backend Services

## Service Architecture

```
AppService (orchestrator)
‚îú‚îÄ‚îÄ PlaybackController (audio engine + queue navigator)
‚îÇ   ‚îú‚îÄ‚îÄ Transport, volume, modes, gapless transitions
‚îÇ   ‚îî‚îÄ‚îÄ reset_next_track() on mode toggles
‚îú‚îÄ‚îÄ Backend services composed on AppService:
‚îÇ   AuthGateway, AlbumsService, ArtistsService, SongsService,
‚îÇ   QueueService, SettingsService
‚îú‚îÄ‚îÄ Orchestrator borrow-handles (built per-call from `app.queue_orchestrator()`
‚îÇ   / `app.library_orchestrator()`, not stored on AppService):
‚îÇ   ‚îú‚îÄ‚îÄ `QueueOrchestrator` ‚Äî five queue-mutation verbs (`enqueue`,
‚îÇ   ‚îÇ   `enqueue_and_play`, `insert_at`, `play_next`, `play`), plus the
‚îÇ   ‚îÇ   `QueueVerb` / `StartPosition` enums consumed by the private
‚îÇ   ‚îÇ   `AppService::dispatch(SongSource, QueueVerb)`. Every public
‚îÇ   ‚îÇ   `play_*` / `add_*` / `insert_*` / `play_next_*` entity-verb
‚îÇ   ‚îÇ   wrapper on AppService is a one-line delegation to `dispatch`,
‚îÇ   ‚îÇ   which resolves the source, applies the single empty guard
‚îÇ   ‚îÇ   (`SongSource::empty_error_message()`, entity-named + toast-
‚îÇ   ‚îÇ   surfaced), then hands the songs to the matching orchestrator
‚îÇ   ‚îÇ   verb. Add new entity-verb combinations as wrappers over
‚îÇ   ‚îÇ   `dispatch` ‚Äî never as hand-rolled resolve+guard+verb bodies.
‚îÇ   ‚îÇ   (`add_song_to_queue_by_id` is the one special case: find-in-album
‚îÇ   ‚îÇ   + custom error, then `queue_orchestrator().enqueue`)
‚îÇ   ‚îî‚îÄ‚îÄ `LibraryOrchestrator` ‚Äî entity resolution for albums / artists /
‚îÇ       genres / playlists / songs / batch via `SongSource::Batch`,
‚îÇ       unified behind `resolve(SongSource)`
‚îú‚îÄ‚îÄ Album/Artist/Songs services share the `LazyAuthedService<A>` newtype
‚îÇ   (`backend/lazy_authed_service.rs`) for their `tokio::OnceCell` +
‚îÇ   constructor closure pattern ‚Äî added in Group D Lane 2 to keep their
‚îÇ   lazy-init bodies from drifting
‚îú‚îÄ‚îÄ On-demand API services (no backend wrapper) ‚Äî generated by paired
‚îÇ   `native_api_factory!` + `subsonic_api_factory!` macros at the bottom
‚îÇ   of `app_service.rs`. Native: `songs_api()`. Subsonic (factory adds
‚îÇ   `server_url` + `subsonic_credential` from `AuthGateway::server_config()`):
‚îÇ   `genres_api()`, `libraries_api()`, `playlists_api()`, `radios_api()`,
‚îÇ   `similar_api()`. All native construction (the macro, the ad-hoc
‚îÇ   `SongsApiService` sites, `LibraryOrchestrator::resolve_genre`) funnels
‚îÇ   through `AuthGateway::build_native_api(factory)` ‚Äî one auth dance, one
‚îÇ   "Not authenticated" error. `LazyAuthedService::build_authed(factory)`
‚îÇ   layers it for ad-hoc (uncached) construction from a wrapped gateway
‚îú‚îÄ‚îÄ Multi-library selection ‚Äî `active_library_ids: Arc<RwLock<HashSet<i32>>>`
‚îÇ   + `all_libraries: Arc<RwLock<Vec<Library>>>`. Empty selection = "no
‚îÇ   filter" (all libraries). Read-mostly hot path on every `load_*`. Persisted
‚îÇ   to redb under `ACTIVE_LIBRARY_IDS_KEY`; mutators: `toggle_library`,
‚îÇ   `set_active_library_ids`, `refresh_libraries` (server fetch + prune).
‚îÇ   Browse loaders consume the snapshot through `_with_libraries` variants
‚îú‚îÄ‚îÄ Artwork ‚Äî server-only, no client-side persistent cache
‚îÇ   ‚îú‚îÄ‚îÄ `AlbumsService::artwork_client: Arc<reqwest::Client>` (bare reqwest)
‚îÇ   ‚îú‚îÄ‚îÄ Concurrent in-flight fetches capped at 16 via a `Semaphore`
‚îÇ   ‚îú‚îÄ‚îÄ `fetch_album_artwork(art_id, size, updated_at)` builds the URL
‚îÇ   ‚îÇ   and dispatches to `fetch_artwork_by_url`; the retry variant
‚îÇ   ‚îÇ   `fetch_artwork_by_url_with_retry` is the single shared retry impl
‚îÇ   ‚îÇ   (3 attempts, 100/200 ms backoff) sitting in front of Navidrome's
‚îÇ   ‚îÇ   `getCoverArt` throttle. Empty-bytes responses surface as errors
‚îÇ   ‚îÇ   so retries can recover instead of caching a blank handle
‚îÇ   ‚îî‚îÄ‚îÄ Session-scoped Handle reuse via UI's `album_art` (LRU 1024 ‚Äî
‚îÇ       doubled when the 2√ó2 quad thumbnails landed) + `large_artwork`
‚îÇ       (LRU 200) maps, plus per-target `CollageArtworkCache` LRUs
‚îÇ       (100 each) for genre/playlist collages (`src/state/artwork.rs`)
‚îú‚îÄ‚îÄ SSE ‚Äî `data/src/services/navidrome_events.rs::parse_sse_event` parses
‚îÇ   events; the subscription itself runs in the UI crate
‚îÇ   (`src/services/navidrome_sse.rs`) and dispatches
‚îÇ   `Message::LibraryChanged(LibraryChange { album_ids, artist_ids, song_ids,
‚îÇ   playlist_ids, genre_ids, is_wildcard })` ‚Üí ID-anchored slot-list reload
‚îÇ   handled by `update/library_refresh.rs`. Each entity branch reloads only
‚îÇ   when its ID list is non-empty (or the payload is a wildcard); wildcard
‚îÇ   events also re-fetch the multi-library list. Artwork is deliberately NOT
‚îÇ   refreshed from SSE ‚Äî Navidrome fires `library_changed` on every
‚îÇ   play-count bump and the auto-refresh caused a one-frame GPU-upload
‚îÇ   flicker. Cover-art replacements propagate via the paged slot-list reload
‚îÇ   and the user-initiated right-click "Refresh Artwork" path.
‚îî‚îÄ‚îÄ TaskManager ‚Äî real supervisor: `shutdown_all(budget)` fires the shared
    cancellation token and awaits every tracked `JoinHandle` (`JoinSet`)
    within the budget, then aborts stragglers. `request_shutdown()` on
    AppService awaits a 500 ms budget. Status channel still streams
    `TaskStatus` to the UI for notifications. UI-side subscription wrappers
    in `src/services/subscription_slot.rs` route the loop / queue-changed /
    task-status receivers through a shared `SubscriptionSlot<T>` (Notify-
    backed wakeup) instead of byte-identical 100 ms polling wrappers.
```

`backend/` modules:
- Service structs: `app_service.rs`, `playback_controller.rs`, `albums.rs`, `artists.rs`, `songs.rs`, `queue.rs`, `settings.rs`, `auth.rs`, `lazy_authed_service.rs` (the `LazyAuthedService<A>` newtype shared by Albums/Artists/Songs).
- Orchestrators: `queue_orchestrator.rs`, `library_orchestrator.rs` ‚Äî built per-call, not stored on AppService.
- UI-projected view-data only (no service): `genres.rs` (`GenreUIViewData`), `playlists.rs` (`PlaylistUIViewData`).
- Cross-entity helpers in `mod.rs`: `Starable` / `Ratable` / `PlayCountable` traits with shared list-mutation helpers (`update_starred_in_list`, `update_rating_in_list`, `increment_play_count_in_list`) and `flatten_participants()`.
- `AuthGateway::server_config() -> (String, String)` returns the `(server_url, subsonic_credential)` pair under a single lock acquisition. Use this everywhere both halves are needed; never call `get_server_url()` + `get_subsonic_credential()` back-to-back.

`services/api/` per-domain modules: `albums.rs`, `artists.rs`, `songs.rs`, `genres.rs`, `libraries.rs`, `playlists.rs`, `radios.rs`, `similar.rs`, `rating.rs`, `star.rs`, `subsonic.rs`, `client.rs`, plus shared helpers `pagination.rs`, `parse.rs`, `sort.rs`.

`subsonic.rs` owns the shared Subsonic plumbing: `subsonic_post` / `subsonic_post_ok` (form-POST + status policy), `SubsonicEnvelope<T>` + `subsonic_get_envelope(...)` (the single `{"subsonic-response": ...}` declaration ‚Äî typed list endpoints fetch through this helper and keep only a file-private inner struct per endpoint), and `deserialize_one_or_many` (single-object-vs-array quirk). Keep default serde on all wire structs (no `deny_unknown_fields`) ‚Äî additive Navidrome fields must stay no-ops.

## Queue System

`SongPool` (HashMap) + `Queue` ordering (`song_ids` + `order` array + `current_order`). Modules: `services/queue/{mod, navigation, order, write_guard}.rs`.

- **Mutation guard** (`write_guard.rs`): every queue write goes through `QueueWriteGuard`, which forces the caller to pick a commit mode at the end of the borrow ‚Äî `commit_save_all` (full save), `commit_save_order` (order-only), or `commit_no_save` (in-memory only). Each `commit_*` method returns a `NextTrackResetEffect` so a future reorder path can't silently skip the engine's gapless-prep reset. Drop without committing is the intentional safety net: it still runs `clear_queued()`, so `?` propagation or a panic between `write()` and `commit_*` can't leave a buffered next song keyed to pre-mutation order.
- **Navigation typestate** (`navigation.rs`): `peek_next_song()` returns a `PeekedQueue<'_>` guard whose only public consumer is `transition()`. The crate-internal `transition_to_queued_internal` is the actual mutator ‚Äî no other call site can advance the queue without first peeking.
- Shuffle off: identity order. Shuffle on: Fisher-Yates with the current song anchored at index 0.
- Every queue mutation calls `clear_queued()` to invalidate the buffered next song.
- All queue mutators ‚Äî mode toggles (`toggle_shuffle`, `set_repeat`, `toggle_consume`), reorders (`move_item`, `insert_*`, `remove_*`, `sort_queue`, `shuffle_queue`), and full replacements (`set_queue`, `add_songs`, `reposition_to_index`) ‚Äî return `NextTrackResetEffect`. The `#[must_use]` discharge contract makes engine gapless-prep invalidation a compile-time obligation across every reorder path, closing the queue/engine desync window that a shuffle + crossfade reorder used to open.
- Progressive build: everything already loaded in the songs `PagedBuffer` plays immediately; a recursive `ProgressiveQueueAppendPage` chain (`src/update/progressive_queue.rs`) fetches the rest at `library_page_size` per page (100/500/1000/5000, default 500).
- Serialization: bincode `Encode` / `Decode` (~3√ó faster than JSON) via `StateStorage::save_binary()` / `load_binary()` ‚Äî no JSON fallback path.
- **Reshuffle on repeat wrap**: shuffle + repeat-playlist re-shuffles the order array when the queue wraps back to the start.

## Batch Operations

`BatchPayload` + `BatchItem` (`data/src/types/batch.rs`) ‚Äî multi-selection batch processing for queue add, playlist add, context menu actions. `BatchItem` variants: `Song`, `Album`, `Artist`, `Genre`, `Playlist`. Built in visual top-to-bottom order via `evaluate_context_menu()` resolved indices.

## SettingsService & SettingsManager

`SettingsService` (`backend/settings.rs`) is a thin async wrapper around `SettingsManager` (`services/settings.rs`). The wrapper's pure pass-throughs are generated by file-private `delegate_setter!` / `delegate_getter!` macros ‚Äî only methods that need a cast, multi-arg signature, or two-call-under-one-lock sequence stay inline.

`SettingsManager` owns `settings: UserSettings` (= `PersistedPlayerSettings` + `ViewPreferences` + `HotkeyConfig`) + `StateStorage`. Loads `config.toml` ‚Üí merges with redb ‚Üí `UserSettings`; the UI-facing live union `LivePlayerSettings` is projected on demand via `get_player_settings()`. Per-field setters persist atomically. `reload_from_toml()` for hot-reload.

`LivePlayerSettings` is the live in-memory union of every user-tunable knob. Both it and `PersistedPlayerSettings` are emitted from ONE `player_settings_schema!` field table in `data/src/types/settings.rs` (M4) ‚Äî a field exists in both structs (or the declared one) from a single row; the divergences (`light_mode` persist-only, `visualizer` live-only, three f64|f32 splits) are declared per row, and the M4 golden-bytes tests pin the persisted wire (field order, serde attrs, default values). `crate::types::player_settings` re-exports the Live twin, and its per-domain submodules (`appearance`, `artwork`, `library`, `navigation`, `playback`, `slot_list`, `strip`, `verbose`, `visualizer`) still own the knob enums. `TomlSettings` is the on-disk shape of `[settings]`. Notable knobs include `font_family`, `library_page_size`, `artwork_resolution`, `volume_normalization` + ReplayGain, per-view column flags (`{view}_show_*`), artwork column, tray, nav, `slot_row_height`, `track_info_display`, default-playlist, strip. Read the structs for the full set.

The 50 per-view column flags live on one canonical `ViewColumns` struct (`data/src/types/view_columns.rs`) embedded as `view_columns` in all three settings structs ‚Äî `#[serde(flatten)]` on `PersistedPlayerSettings` / `TomlSettings` keeps every `{view}_show_*` key flat on both wire formats (pinned by the `*_column_keys_stay_flat_*` tests), and its single hand-written `Default` is the source of truth for shipped column defaults (so `LivePlayerSettings::default()` carries real column defaults, not all-false). Field names inside `ViewColumns` are wire keys ‚Äî renaming one requires a migration.

### `define_settings!` registration

Every setting backed by the Settings UI is registered via the `define_settings!` macro (`data/src/services/settings_tables/{general,interface,playback}.rs`; the macro itself lives in `data/src/types/setting_def.rs`). Each entry declares a key, label, scalar/array type, default, on_dispatch hook, ui_meta cluster, and a `write` closure that round-trips the live value back into TOML (the `write` arm was added 2026-05-17 ‚Äî keep both directions on every key). The macro emits the dispatch arm + the per-tab `dump_<tab>_tab_player_settings` helper. Use `SettingsSideEffect` (`data/src/types/settings_side_effect.rs`) variants to thread side effects (toasts, atomic flag flips, library reloads) back to the UI from `on_dispatch`. Hotkey actions follow the same single-table pattern via `define_hotkey_actions!` in `data/src/types/hotkey_config/action.rs`. Per-tab `{General,Interface,Playback}SettingsData`, `SettingItem`, `SettingMeta`, `SettingValue`, and `SettingsEntry` live in `data/src/types/`.

Adjacent enum macros: `define_labeled_enum!` (`data/src/types/labeled_enum.rs`) for label/wire-string-backed enums (the `player_settings` knobs ‚Äî `LibraryPageSize`, `NavLayout`, `TrackInfoDisplay`, `StripClickAction`, `IconSet`, `VerboseConfig`, etc.; `QueueSortMode` and `VisualizationMode` keep their own hand-rolled metadata tables); `atomic_u8_enum!` (`src/atomic_u8_enum.rs`) for the twelve enum-typed `UiModeFlags` atomics (eleven in `theme/ui_mode.rs` + `RoundedMode` in `theme/radius.rs`) ‚Äî bytes derive from declaration discriminants (`Variant as u8`, list variants in data-crate declaration order) and are a transient in-process cache encoding only; persistence goes through the serde wire strings. Both keep unknown input from silently falling through (label fallback / defensive byte fallback).

## Theme System

- `ThemeFile`: `name`, `dark: ThemePalette`, `light: ThemePalette`
- `ThemePalette`: `BackgroundConfig` (6 levels), `ForegroundConfig` (5 levels), `AccentConfig`, four `SemanticColorConfig` (danger / success / warning / star), `VisualizerColors`, plus a serde-default `border` hairline color (empty string ‚Üí runtime derives it by darkening `background.hard` 30 %)
- `config.toml` stores `theme = "name"` ‚Äî points to `~/.config/nokkvi/themes/{name}.toml`
- Font is a **global setting** (`font_family` in `LivePlayerSettings` / `TomlSettings`), decoupled from `ThemeFile`

## Domain Types (`data/src/types/`)

Iced-free. Key types:
- `PagedBuffer<T>` ‚Äî replaces `Vec<T>` for library data. `Deref<Target=[T]>`. **`generation()`** monotonic counter bumps on every mutation (use for `(query, generation)` filter-cache keys)
- `HotkeyConfig` ‚Äî HashMap with O(1) lookup
- `LivePlayerSettings`, `TomlSettings`, `TomlViewPreferences`
- `Queue`, `QueueSortMode` (physical sort: Album/Artist/Title/Duration/Genre/Rating/MostPlayed/Random ‚Äî Random re-rolls on re-select)
- `NextTrackResetEffect` (`next_track_reset.rs`) returned from every queue mutator (mode toggles + reorders + replacements) to chain engine `reset_next_track()` calls. `apply_to(&Mutex)` locks the engine internally; `apply_locked(&mut Engine)` is for callers already holding the engine lock (completion callback consume path, `play_next` / `play_previous`).
- `SongPool`, `BatchPayload` / `BatchItem`
- `LibraryFilter` ‚Äî ID-based cross-view navigation filter
- `PlaylistEditState` ‚Äî dirty detection
- `InfoModalItem` ‚Äî owned data for the Get Info modal
- `ReactiveProperty<T>` / `ReactiveVecProperty<T>` (`reactive.rs`) ‚Äî thread-safe shared property with subscribe-on-change

## API Patterns

- **Star**: optimistic UI + revert on failure
- **Rating**: +/- hotkeys
- **Playlist CRUD**: Navidrome-native REST (not Subsonic) for writes
- **MPRIS**: D-Bus background task; full metadata for both queue songs and radio streams
- **Tray**: ksni-based StatusNotifierItem in `src/services/tray.rs`; emits `TrayEvent` to the UI
- **Internet Radio**: Subsonic `getInternetRadioStations` + mutative CRUD tasks
