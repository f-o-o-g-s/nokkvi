# Audio Engine

## Architecture

```
CustomAudioEngine
├── AudioDecoder (Symphonia)
│   ├── Standard: HTTP via RangeHttpReader (256KB chunks, MAX_CACHED_CHUNKS=16, prefetch on seek + background sliding-window read-ahead)
│   └── Radio:    HTTP via AsyncNetworkBuffer (tokio async → bounded mpsc → sync Read) with generation-gated auto-reconnect. The EOF reconnect loop is the free async `radio_reconnect_loop()` → `RadioReconnectOutcome::{Reconnected, Superseded, GaveUp}` control token (the caller owns the `continue 'decode_loop` / `break` flow); the jitter prebuffer is the free `radio_jitter_prebuffer_step()`
├── AudioRenderer (ring buffers) → visualizer callback from StreamingSource
│   └── RodioOutput (shared rodio Mixer) → ActiveStream per track
│       └── StreamingSource (rodio::Source) → EqProcessor → lock-free ring buffer → pipewire callback
├── CrossfadeCoordinator (engine, `engine.rs`) — owns the engine-side crossfade cluster: the `CrossfadePhase` + the `enabled` / `duration_ms` config + the bit-perfect / min-track / album-continuity / offset / bar-snap / skip-silence mirrors + the per-transition state: `suppress_this_transition` (M4 policy flag), `duration_override_ms` (M8 bar-snap override), `skip_fade` (M7 manual-skip marker). `crossfade_eligible()` (shared coarse arm gate), `effective_duration_ms()` (the ONLY sanctioned duration read on arm/trigger/fire paths — reading `duration_ms` directly silently misses the override on the seek-rearm / EOF-fallback paths), and `is_crossfade_live(&renderer)` (engine-not-Idle OR renderer-Active reconciliation for `reset_next_track`) live on it. Does NOT own the renderer `CrossfadeState` nor the `crossfade_duration_shared` atomic (a decode-loop channel)
│   └── CrossfadePhase enum: Idle → Active { decoder, incoming_source } → OutgoingFinished { decoder, incoming_source }
├── CrossfadeState enum (renderer): Idle / Armed / Active — per-phase data lives inside the variant (incl. the fade `curve` captured at `start_crossfade`); `mem::replace` swaps phases atomically
├── TransportFade enum (renderer, M5/M6): Idle / FadingOut{Pause|Stop} / FadingIn / SwitchFadeIn — short per-stream volume ramps for pause/resume/stop and the radio-switch fade-in. SEPARATE from `CrossfadeState` by design; engages only while the crossfade machine is Idle (see "Transport & Switch Fades")
├── FadeCoordinator (engine, `engine.rs`) — engine-side mirror of exactly the Fading knobs the engine consumes: the stop pair (`fade_on_stop`/`fade_stop_ms`), `fade_radio_transitions` (M6), the skip pair (`fade_on_skip`/`fade_skip_ms`, M7). The PAUSE pair lives on the renderer (its two consumers — `begin_pause_fade`, the resume fade-in in `start()` — are there): each side keeps exactly what it reads
├── PreparedTransitionDirectives — per-transition verdicts (`suppress_crossfade`, `duration_override_ms`, `gap_offset_ms`) the CONTROLLER computes at gapless-prep time from `Song` metadata (which never crosses the engine boundary) and threads into `store_prepared_decoder`; all share the M4 suppress-flag lifecycle (applied by `apply_transition_directives` AFTER the store's internal `reset_next_track`, cleared BY `reset_next_track`, re-derived on every prep)
├── DecodeLoopChannels (engine, `engine.rs`) — bundles the lock-free atomics the primary decode loop reads/writes (`source_generation`, `decoder_eof`, `stream_is_infinite`, `crossfade_duration_shared`, `live_bitrate`, `skip_fade_pending`, `gap_offset_ms`); `clone_for_decode_loop()` hands the spawned task exactly those Arcs by identity-preserving clone (an `Arc::ptr_eq` test guards the same-Arc invariant). `live_sample_rate` is intentionally excluded — engine-only, never cloned into the loop
├── GaplessSlot — bundles `decoder` + `source` + `prepared` + `replay_gain` under one `tokio::Mutex` so the decode loop, async path, and `cancel_crossfade` always lock together (audit IG-13). The `replay_gain` rides WITH the slot so a prep landing mid-blend never clobbers the live blend's `pending_crossfade_replay_gain`; every slot consumer re-stages it. The inline EOF swap is extracted into the free async `try_gapless_swap()` → `GaplessSwapOutcome::{Swapped, NotPrepared, FormatMismatch, CrossfadeActive, SkipFadePlanPending, DecoderMissing}`; `bump_for_gapless` is its SOLE decode-loop bump
├── DecodeLoopHandle / SourceGeneration (`generation.rs`) — typed atomic-counter wrappers; `bump_for_user_action` / `bump_for_gapless` / `accept_internal_swap` make every "stop the loop" / "invalidate callback" call site self-documenting
├── LiveStringSlot (`engine.rs`) — `Arc<RwLock<Option<String>>>` newtype for the engine's hot-path live strings (`live_icy_metadata`, `live_codec_name`). Writes go through `reset()` (non-blocking `try_write`) / `set()` (blocking) — the type encodes the reset-side discipline (B11 hot-path) so it can't drift into a blocking `write()`. Attribution of live ICY to a station goes through `is_playing_source(url)` (`self.source == url`; `self.source` is set at the END of `set_source`): during a station switch the UI's `active_playback` flips synchronously but `set_source` runs async after `stop()`, so for a window the engine still streams — and reports ICY for — the PREVIOUS source; gate any ICY-derived state (now-playing art/title, radio scrobbling) on this check
└── EqState (eq.rs) — shared atomic gains passed to each StreamingSource
```

## Audio Output (native PipeWire)

One native PipeWire stream via a shared `rodio::Mixer`:
- `SfxEngine` owns the app-wide `ActiveSink` (`NativePipewire` or `Cpal` fallback)
- `NativePipeWireSink` runs a dedicated `pw_nokkvi_out` thread with its own PipeWire mainloop
- Each track gets an `ActiveStream`: `RING_BUFFER_CAPACITY = 5_000_000` samples (~52 s at 48 kHz stereo, sized for radio jitter) + a `StreamingSource` added to the mixer
- `StreamingSource` implements `rodio::Source` (pull model) — pipewire callback pulls f32 samples

## Critical Rules

- **Codec registry**: every Symphonia decoder/lookup MUST go through `audio::symphonia_registry::codecs()`, never `symphonia::default::get_codecs()`. The shared registry adds `symphonia-adapter-libopus` on top of the Symphonia defaults; the upstream default registry has no Opus decoder (pdeljanov/Symphonia#8 open since 2020), so any direct call to `get_codecs()` re-breaks `.opus` playback (see GH#3). The `audio::symphonia_registry::probe_and_make_decoder(mss, hint, enable_gapless)` helper owns the shared probe+decoder build; `enable_gapless` is load-bearing — `true` only for the primary init in `AudioDecoder::open_input`, `false` for the `ResetRequired` reprobe and `SfxEngine::decode_wav_stream`.
- **Track changes**: create fresh decoders **before** locking the engine; release the engine lock during decoder operations. Use `engine.load_track_with_rg(url, rg, expected_duration_ms)` — the atomic pair that stashes ReplayGain on the renderer and then calls `set_source(url, expected_duration_ms)`, replacing the historical `set_pending_replay_gain` + `load_track` / `set_source` pairing in `PlaybackController`. `expected_duration_ms` carries the server's track length into the decoder for the probed-duration cross-check (see "Probed Duration").
- **`SourceGeneration`**: typed atomic counter; `bump_for_user_action()` on every user-driven source change. The renderer snapshots `current()` before releasing the engine lock and discards stale completion callbacks.
- **Next-track reset**: `reset_next_track()` clears the prepared decoder, disarms crossfade, and clears every per-transition directive — the suppress flag, the bar-snap `duration_override_ms` (restoring `crossfade_duration_shared` to the global), and the pending gap. Every queue mutator (mode toggles, move/insert/remove/sort, set_queue, add_songs, reposition_to_index) returns `NextTrackResetEffect` — a `#[must_use]` token that the caller dispatches via `apply_to(&engine)` (engine mutex) or `apply_locked(&mut engine)` (engine lock already held). The token makes the reset a compile-time obligation, so a new reorder path can't reintroduce the shuffle + crossfade UI-vs-engine desync.
- **Track-completion path**: the playback navigator releases its lock across engine I/O — do not re-introduce a held lock around `PeekedQueue::transition()` / `set_source()`.
- Decoupled render thread: 20 ms intervals (50 Hz), handles crossfade tick + completion detection.

## Network Resilience (issue #9)

Layered subsystems keep finite HTTP streams playing on shallow networks (shipped v0.6.9; hi-res deadlock fixed v0.7.1):

- **Background read-ahead** (`range_http_reader.rs`): `prefetch_loop` keeps a sliding window of `PREFETCH_WINDOW_CHUNKS = 5` chunks ahead of the read cursor, topping up when the lead drops to `PREFETCH_LOW_WATERMARK_CHUNKS` (= 2). Spawned lazily by `maybe_spawn_prefetch()` on the first forward-sequential read (current chunk == previous + 1), so the FLAC init-probe's binary-search seeks never spawn a task chasing a bouncing cursor. A const assert (`1 + PREFETCH_WINDOW_CHUNKS + EXPECTED_BEHIND_CHUNKS <= MAX_CACHED_CHUNKS`) guarantees the LRU can never evict the live window — keep it satisfied when resizing any of the three. The synchronous seek-time prefetch is a separate, unchanged path.
- **Decode cushion** (`engine.rs`): the decode loop buffers `CUSHION_MS = 1100` ms ahead (TIME-based, scaled by frame rate, grown for long crossfades); backpressure releases at `CUSHION_MS / BACKPRESSURE_RELEASE_DIVISOR` (= 3).
- **Pause-and-rebuffer** (`renderer.rs`): pure state machine `rebuffer_action()` → `RebufferAction::{Enter, Hold, Exit, None}` driven from `render_tick`. Watermarks are TIME-based — enter below `REBUFFER_ENTER_MS = 200`, resume at `REBUFFER_RESUME_MS = 800` — converted to samples per-format via `frame_rate`. Fixed-SAMPLE budgets were the v0.6.9 hi-res deadlock: at 96k/192k they shrank in time until rebuffer entry sat above the decode backpressure-release point, freezing the ring. Compile-time interlocks encode the ordering (`REBUFFER_ENTER_MS * BACKPRESSURE_RELEASE_DIVISOR < CUSHION_MS`, `REBUFFER_RESUME_MS < CUSHION_MS`, `REBUFFER_ENTER_MS < REBUFFER_RESUME_MS`) — keep all three satisfied when changing any of these constants. Finite (seekable) streams only: never radio, never mid-crossfade. `MAX_REBUFFER_TICKS = 500` is the dead-socket safety valve.
- **Latch discipline**: `reset_rebuffer_latch()` zeroes `rebuffering` / `rebuffer_primed` / `rebuffer_ticks` on every fresh-ring transition — start, stop, seek, and `finalize_crossfade` (carrying a stale `rebuffer_primed` across a sample-rate change is the one path that re-creates the deadlock; test `finalize_crossfade_resets_rebuffer_latch`). `pause()` deliberately clears only `rebuffering`. The latch primes once the ring first reaches the resume target, so a cold track start cannot false-pause at 0:00.

## Volume & Fade Application

Dual-path user volume: PipeWire native (preferred) or software fallback.

- **PipeWire-native** (`pw_volume_active = true`): software volume at 1.0; SfxEngine sends cubic (`v³`) to PipeWire over IPC.
- **Software fallback**: exponential amplitude curve (`perceptual_volume`), per-sample exponential smoothing (~5 ms EMA) for anti-aliasing.

The fade is a **separate channel from user volume**: `StreamHandle.fade_coeff` atomic + a `smoothed_fade` EMA in `StreamingSource`, applied **linearly** on BOTH arms — non-bit-perfect output is `sample · smoothed_perceptual(user_vol) · smoothed_fade`; bit-perfect is `sample · smoothed_fade` with the `UNITY_SNAP_EPSILON` snap at `fade_coeff ≥ 1.0`. Never route a fade through the `volume` atomic: user volume is perceptually re-curved (`perceptual_volume`), and re-curving the fade coefficient was THE historical crossfade bug — every default-path fade midpoint realized `perceptual(0.5)` ≈ −30 dB per stream, a near-silence hole mid-blend.

- `StreamingSource::new(…, initial_fade, …)` seeds BOTH the `fade_coeff` atomic and `smoothed_fade` from `initial_fade`: 0.0 for the crossfade incoming stream, 1.0 for fresh play/seek streams. Hardcoding the smoother at 1.0 plays the first ~5 ms of every crossfade incoming at near-full gain (onset burst).
- `tick_crossfade` writes `fade_gains(curve, progress)` (`audio/fade_curve.rs`) to both streams' `fade_coeff`; the curve is captured into the renderer's `Active` variant at `start_crossfade` (a mid-fade settings change never tears an in-flight blend): **Equal Power** (default — `cos`/`sin`, power-sum 1, flat loudness for uncorrelated tracks), **Constant Gain** (the historical `cos²`/`sin²`, amplitude-sum 1 — equal-power only for correlated content, ~3 dB midpoint dip otherwise), **Linear**. Fade-in endpoints are exactly 1.0 — the unity snap depends on it.
- `fade_coeff → 1.0` resets at BOTH `finalize_crossfade` and `cancel_crossfade`; in `cancel_crossfade` the reset is **unconditional, outside the `!self.paused` guard** — resume (`start()`) restores only `volume`, so a paused-path cancel would otherwise strand the restored outgoing at reduced gain.
- A mid-crossfade `set_volume` pushes `stream_volume()` to BOTH streams' `volume` atomics **unconditionally** (the tick writes only `fade_coeff`; gating the push on `!paused` would permanently lose a volume change made while paused mid-blend).
- **De-click onset ramp** ("Smooth Track Starts", default on): non-bit-perfect streams seed the user-volume smoother at 0 (`smoothed_volume: if smooth_starts && !bit_perfect { 0.0 } else { volume }`) so the existing EMA ramps 0 → target over ~23 ms from the first sample. The volume EMA advances on **real samples only** — clocking it on ring-starvation fills burns the ramp during the decoder's fill latency and ships the click anyway. `smoothed_fade` stays pull-clocked (the fade envelope is renderer-wall-clock-driven). Bit-perfect streams keep an instant, honest onset.

## Transport & Switch Fades

`TransportFade` (renderer): `Idle / FadingOut{Pause|Stop} / FadingIn / SwitchFadeIn` — short per-stream volume ramps (20–500 ms, default 100 ms) replacing the mid-waveform atomic flips on pause/resume/stop, ticked by the 20 ms render thread. Kept SEPARATE from `CrossfadeState`; `transport_fade_engageable()` refuses while a crossfade is armed/active (never fight a blend), on a bit-perfect stream (its arm reads only `fade_coeff`, so a volume ramp is an inaudible delay; ramping `fade_coeff` outside a crossfade would break bit-identity), and on a drained ring (nothing audible to fade). "Fade on Pause / Resume" and "Fade on Stop" both default OFF.

- **Guard-lift**: `begin_pause_fade`/`begin_stop_fade` set `self.paused = true` IMMEDIATELY (freezing the completion gate, the rebuffer path, and the armed crossfade trigger below `render_tick`'s early-return); `tick_transport_fade()` runs ABOVE that early-return. Deferring `paused` until ramp completion would let a pause near end-of-track advance the queue mid-ramp.
- **Settle tick**: the ramp-floor tick writes volume 0.0, sets `settling`, and returns; the NEXT tick applies the real `pause()`/`silence_and_stop()`. Applying the end action in the same tick cuts at the source EMA's pre-floor gain — up to a full-amplitude click at the 20 ms minimum.
- A per-fade **generation token** discards stale completions; an interrupting ramp seeds `from` off the stream's last-written `volume` atomic so the level never snaps.
- `engine.stop()` runs the out-ramp with a bounded wait (ramp + 250 ms deadline) BEFORE `stop_render_thread()` — the render thread is what drives the ramp. Stop-from-paused skips the ramp entirely (the early-return already froze the stream). `set_source`'s internal stop calls `stop_without_fade()`: track changes belong to "Fade on Skip", never to "Fade on Stop".
- `renderer.init()`'s gapless-reuse branch restores the reused primary's `volume` when it abandons an in-flight ramp — the one cancel site that neither restores the volume nor rebuilds the stream.
- **Radio-switch fade** ("Fade Radio Switches", default off): both switch edges run a fixed `RADIO_SWITCH_FADE_MS` (250 ms) out-ramp via the shared `run_bounded_out_fade`, then arm a one-shot fade-in request consumed at the next fresh stream build (in `init()`, before `play()`'s prebuffer can feed the mixer a full-gain sample). Edge detection differs: queue→radio is caller-known — the UI radio-start paths call `engine.stop_for_radio_switch()` (flag off ⇒ delegates to `stop()`, byte-identical to the historical explicit stop); radio→queue is engine-known (`stream_is_infinite` still reflects the outgoing at `set_source` entry). Finite→finite track changes stay instant.
- `SwitchFadeIn` is **first-audio-triggered, not wall-clock**: it holds the fresh primary silent until the mixer pulls real samples past a `baseline_samples` snapshot, then ramps 0 → target (a wall-clock ramp would finish during the radio prebuffer silence and pop at full gain when audio arrives). `pause()` **re-arms** it with a fresh baseline instead of cancelling — the radio jitter-prebuffer dance re-pauses repeatedly, and the pre-jitter trickle of real samples must not silently burn the ramp; `start()` leaves it alone. A bit-perfect fresh stream drops the request.
- **Boundary skip fade** ("Fade on Skip" = Boundary Fade, also the blocked-blend fallback): `run_skip_out_fade` delegates to the same bounded out-ramp with the skip duration, then the normal load proceeds. No separate `TransportFadeTarget::Skip` variant exists — its end action would be identical to `Stop`.

## Volume Normalization & ReplayGain

`VolumeNormalizationMode`: `Off`, `Agc`, `ReplayGainTrack`, `ReplayGainAlbum`. Settings live in the Settings view's Playback tab, "Volume Normalization" section (`src/views/settings/items_playback.rs`): mode dropdown, AGC target level (AGC only), and — in RG modes — preamp dB, fallback dB, fallback-to-AGC, prevent-clipping. Renderer reads `volume_normalization_mode` and resolves a gain factor; `RodioOutput` applies it via `source.amplify(gain).limit(LimitSettings::dynamic_content())`. Primary loads stash incoming-track ReplayGain via `load_track_with_rg(url, rg, expected_duration_ms)`; the crossfade decoder uses `set_pending_crossfade_replay_gain()` before its stream is built — both paths land the right factor at stream creation.

## Bit-Perfect

`BitPerfectMode` (`types/player_settings/playback.rs`): tri-state `Off` / `Strict` / `Relaxed` (legacy bool records deserialize `true → Strict`, `false → Off`). The engine mirrors the setting in `crossfade.bit_perfect_mode` (the `CrossfadeCoordinator` field that `crossfade_eligible` reads), kept in sync by `set_bit_perfect`, which also delegates to `renderer.set_bit_perfect`.

- **Active condition**: `bit_perfect_active() = mode.builds_bit_perfect() && pw_volume_active` — both `Strict` and `Relaxed` build bit-perfect; it requires PipeWire-native volume so no software gain touches the samples.
- **DSP bypass** (`StreamingSource::next`): bit-perfect skips EQ entirely and, at unity, returns the raw decoded sample untouched (no software volume, not even the unity-curve multiply) so a settled body stays bit-identical. During a Relaxed crossfade the `fade_coeff` atomic carries the fade coefficient, applied directly (already shaped by the tick's selected curve — not re-curved through `perceptual_volume`).
- **Honest badge**: `current_stream_bit_perfect` is captured at stream-build time, not read live, so a mid-track toggle can't claim BIT-PERFECT for a stream still on the DSP path.
- **Crossfade gate**: `crossfade_blocked(current, incoming)` returns `Strict` → always blocked, `Relaxed` → blocked only on sample-rate / channel-count mismatch (the DAC can't re-clock mid-blend), `Off` → never. Both crossfade triggers gate on this one method with the same `(current, incoming)` pair.
- **Fades & trims stand down**: bit-perfect streams refuse transport/switch/boundary fades (see "Transport & Switch Fades"), and BOTH silence-trim halves (leading trim and the trailing early trigger) stand down under Strict AND Relaxed — gated on `builds_bit_perfect()` (mode intent, not live `pw_volume_active`, so the verdict can't flap with the output backend). Dropping or never playing decoded samples is a content change.

## Probed Duration

`sanitize_probed_duration(probed_ms, expected_ms)` (`decoder.rs`) cross-checks Symphonia's probed length against the server's track length (threaded in as `expected_duration_ms`). When the probe wildly overshoots the server metadata (the Symphonia#516 Xing/CRC bug reports ~30 min for a ~4 min MP3), the server value wins. `compute_seek_scale(probed_ms, sanitized_ms)` then derives `seek_scale` (1.0 unless the probe was overridden) so seeks land at the right wall-clock position against the corrected timeline.

## Equalizer

- 10-band graphic EQ: `EqState.gains: Arc<[AtomicU32; EQ_BAND_COUNT]>` (`eq.rs::EQ_BAND_COUNT = 10` is the single source of truth for every gain-array width, including the `eq_gains` fields on the settings/types side), `EqProcessor` per-stream biquad bank
- Bands: 31 Hz–16 kHz (ISO standard center frequencies), ±12 dB clamp. Selecting a preset auto-enables the EQ.
- Headroom: −1 dB applied only when the max boost > 0 dB.

## Crossfade

Two concurrent `ActiveStream`s. Guards: both durations known (non-zero), both songs ≥ the configured minimum track length (`crossfade_min_track_secs`, 0–60 s, default 10 s), duration clamped to `min(xfade, shorter / 2)`. A controller-side policy verdict (`audio/crossfade_policy.rs::crossfade_decision`, computed at gapless prep from Song metadata — min-track floor + the opt-in same-album continuity gate `crossfade_album_gapless`) rides down as a per-transition suppress flag on the `CrossfadeCoordinator`: set by `store_prepared_decoder` AFTER its internal `reset_next_track` and BEFORE the arm, cleared by `reset_next_track`, and gated at BOTH trigger sites (`arm_renderer_crossfade`, which also covers the rearm-after-seek path, and `try_start_crossfade_transition`) so a suppressed transition falls through to the gapless join. The position-based trigger in `render_tick` must be **synchronous** — `mem::replace` swaps `crossfade_state` from `Armed` to `Active` in the same tick as the position check, then signals the engine async. Without the synchronous state flip, EOF fires first → hard cut. Cancellation splits by state: `cancel_crossfade()` clears `Active` (the in-flight stream); `disarm_crossfade()` clears `Armed` (metadata only). Pair them when both must clear (engine-level `cancel_crossfade`, renderer `stop`). `renderer.seek` calls only `cancel_crossfade()` so a pending gapless `Armed` state survives the seek and the position-based trigger can still fire.

Eligibility is the shared `CrossfadeCoordinator::crossfade_eligible()` predicate (`enabled || bit_perfect_mode == Relaxed` — the Relaxed gate lets same-format tracks blend under bit-perfect; see "Bit-Perfect") that all three arm/trigger sites (`store_prepared_decoder`, `rearm_crossfade_if_prepared`, `try_start_crossfade_transition`) call; the `crossfade_blocked()` gate then vetoes ineligible format pairs. **Finalize before EOF**: `try_finalize_crossfade(is_eof, renderer_fade_active)` finalizes the `Active` phase as soon as the renderer reports its fade is no longer active, even when the outgoing decoder has **not** hit EOF — past the fade the outgoing is at zero volume, so its remaining tail is inaudible and is discarded. Requiring `is_eof` for the `Active` phase left a torn state for seconds on coarse VBR seeks (renderer promoted the incoming, engine still `Active`), then promoted an already-EOF decoder so the next track was instantly skipped. The decode loop's `decoder_eof` signal is generation-gated so a loop superseded mid-`read_buffer` (which can block for seconds) can't store EOF for a no-longer-current track.

**Manual-skip crossfade** ("Fade on Skip" = Crossfade): three phases across the lock boundary (invariant: no engine lock held across decoder creation). Phase 1, `plan_skip_fade` (navigator, UNDER the locks) fully sequences the skip — cursor, history, consume — cancels the pre-skip transition via `reset_next_track` (nothing can finalize during the build), bumps `bump_for_user_action`, and latches the generation-keyed `skip_fade_pending` window. Phase 2, controller `complete_skip_fade` builds the incoming decoder with NO locks held (`AudioDecoder::default() + init()`, registry path). Phase 3, `engine.crossfade_to_next` re-locks and fires the blend `Active` **directly** — there is no `Armed` hop for an on-demand fire; the invariant preserved is renderer-Active-before-engine-phase + cancel-live-first, not the `Armed→Active` swap. The direct fire bypasses `arm_crossfade`, so its gates are re-applied (`crossfade_blocked`, known durations, min-track floor, `shorter/2`) plus a remaining-audio clamp (`skip_fade_duration_ms` — a fade must never outlive the outgoing). Supersession is two-fold: `skip_fade_seq` (only the latest skip completes) and a `source_generation` snapshot taken post-plan-bump (any competing user action wins; the fire bumps AGAIN, closing the window). While the window is open, completions defer at `on_renderer_finished`, the inline gapless swap stands down (`GaplessSwapOutcome::SkipFadePlanPending`), and `store_prepared_decoder` takes its store-WITHOUT-reset/arm branch (a mid-blend/mid-window store must neither cancel the live blend nor arm against the already-advanced cursor; `finalize_crossfade_engine` re-arms from the slot). The `skip_fade` marker makes a skip-fade finalize suppress the completion callback (the queue already advanced at skip time — firing `decide_transition` would advance it twice) and the `last_transition_was_crossfade` stamp; `recover_stalled_crossfade`'s skip-aware branch hard-loads the skip target instead of routing through the end-of-track machinery. The `Blocked`/failure fallback re-checks `immediate_playing()`: after a stop/pause during the build (neither bumps the generation) it stages the target via `load_track_with_rg` WITHOUT `play()`, so a later Play starts what the queue names. The CLICK paths (queue click / play-from-here / browse-view play — all three funnel through `PlaybackController::load_play_and_set_current`) route through this same three-phase machinery via `plan_click_play` (M10): the click's own queue sequencing replaces `plan_skip_fade`'s navigator half, the pre-gate is `click_skip_crossfade_viable()` (M7 viability + a bit-perfect-Strict short-circuit so Strict clicks stay byte-identical hard cuts), Boundary mode runs the same `run_skip_out_fade` before the hard load, and mode Off / paused / stopped / radio keep today's hard path untouched.

**Content-aware overlap** (all opt-in, default off/0). *Leading trim* ("Skip Silence Between Tracks"): a decoder-side pre-roll scan on PREPARED transition decoders (`LeadingTrim` in `decoder.rs`) drops frame-aligned leading audio below `SOURCE_SILENCE_THRESHOLD` (`audio::mod`, 1e-3 ≈ −60 dBFS peak — one shared constant so the two halves can't disagree) up to a 15 s budget; any decoder `seek()` ends the scan; the scan lives inside the registry-backed decode path. *Trailing trigger*: a lazily-enabled per-stream level meter (`StreamHandle::{level_meter_enabled, recent_source_peak}` — real samples only, so ring starvation can never fake musical silence; seeded loud) lets `tick_trailing_silence` fire the armed blend early after 500 ms of sustained sub-threshold level, only inside a 15 s window before the trigger point (a mid-song quiet passage can never fire). Both halves stand down under bit-perfect modes. *Gap/overlap offset* (`crossfade_offset_secs`, −2…+2 s): negative mirrors to `renderer.crossfade_lead_ms` so the Armed position trigger fires |offset| early; positive rides `transition_prep_cfg()` → per-transition `gap_offset_ms` → the decode loop's EOF silence injection (`inject_transition_gap` — one-shot `swap(0)`, stands down when a blend or an open skip-fade window owns the transition; album-continuity joins zero it controller-side). *Bar-snap* ("Snap Crossfade to Musical Bars"): the controller snaps the fade to whole 4/4 bars of the outgoing's `Song.bpm` (`crossfade_policy::bar_snapped_crossfade_ms`, pure; `Some(0)` BPM guarded), clamped ±1 bar within the 1–12 s slider bounds, and threads it down as `duration_override_ms`. The override follows the M4 suppress-flag lifecycle exactly and is mirrored into `crossfade_duration_shared` in lockstep — the decode cushion must always cover the duration that actually plays.

**Stall watchdog**: `tick_crossfade`'s completion gate at `progress ≥ 1.0` is empty-ring **OR** liveness-stalled. `crossfade_liveness::IncomingLiveness` is a per-fade handle the crossfade decode loop brackets around ONLY its blocking decode/network call (backpressure sleeps, EOF exits, and lock handoffs stay outside the bracket and always read live); one read continuously in flight ≥ `CROSSFADE_STALL_READ_MS` (3 s) counts as stalled, and every race degrades toward "live" (= promote, the pre-watchdog behavior). Never use a count-based discriminator — `crossfade_buffer_count` grows and shrinks for healthy reasons. Recovery routes through the existing `IncomingStalled → on_renderer_crossfade_stalled → recover_stalled_crossfade` path (gated on `is_crossfade_live`, the 2c645885 livelock guard) and signals exactly once per fade (`stall_recovery_signalled` latch, cleared with the fade at start/cancel/finalize).

## Visualizer Sync

`StreamingSource::next()` → pre-volume samples (S16-scaled, volume-independent) → viz_buffer → FFT thread (60 fps, `try_lock()` only) → display buffers (Mutex) → shader GPU. `is_dirty()` gates redraws — GPU idle when paused. Toggling the visualizer **off** also releases CPU: a `viz_enabled` atomic (threaded renderer → `RodioOutput` → `StreamingSource`, ANDed with `feeds_visualizer`) skips the per-sample tap, and a synchronous `feed_active` flag early-returns the 60 Hz FFT worker — both driven from the cycle-visualization handler and synced at login so a persisted "off" never spins the worker. Only the active primary feeds the shared callback: `StreamHandle.feeds_visualizer: Arc<AtomicBool>` is set `true` at construction for primary streams (init / seek-recreate) and `false` for the crossfade incoming stream. `tick_crossfade` hands the feed off to the incoming at the fade midpoint (`progress >= 0.5` — where every curve's gains cross, so the handoff is curve-independent) so the viz never freezes when the outgoing's ring drains during the crossfade tail; `finalize_crossfade` then reaffirms `true` on the newly-promoted primary, and `cancel_crossfade` restores `true` on the restored outgoing (a cancel landing after the midpoint handoff would otherwise leave the spectrum frozen until the next track). Without these gates two concurrent streams at different rates would each tag batches with their own rate, flipping the visualizer's stored-rate atomic every batch and thrashing the spectrum engine into constant reinit (blank bars during cross-rate crossfade windows); without the midpoint handoff the visualizer freezes once the outgoing decoder EOFs and its ring drains, until finalize.
