Commit only the files you (Claude) directly Edited or Wrote in this conversation, and fold any needed `## [Unreleased]` changelog update into that same commit. Leave everything else alone — pre-existing modifications, files the user changed, files from other agents, untracked files you didn't create.

## Steps

1. Run `git status` and `git diff` in parallel to see all working-tree changes.
2. Build a list of files you touched in this session by scanning your own prior `Edit` and `Write` tool calls. That exact set is what you stage. New files created via `Write` belong on the list; files modified by anyone else stay off it.
3. If your set is empty, say so and stop — leave the working tree alone. (`CHANGELOG.md` edited below counts as authored this session, so it joins the set.)
4. Match this repo's existing commit style: run `git log --oneline -10` and follow whatever convention you see (e.g. Conventional Commits `type(scope): description`, lowercase, imperative, no trailing period). For nokkvi the allowed scopes are listed in `CLAUDE.md`.
5. Draft the commit message focused on **why**, not what — the diff is the what. One subject line + a short body if non-obvious. End the message at the body. Skip any `Co-Authored-By` or "generated by Claude" trailer.
6. **Refresh the changelog in the same commit.** Update `## [Unreleased]` in `CHANGELOG.md` so it covers every user-visible effect since the last release, including the change you're about to commit. Apply your wording directly — no approval gate, no separate confirmation.
   - The change being committed isn't a commit yet, so describe it from the diff and the message you just drafted, not from `git log`. Also fold in any prior commit since the last release whose user-visible effect isn't yet reflected — find the boundary with `last_version_commit=$(git log --oneline --all --grep='bump version' -1 --format='%H')` and read `git log --no-merges --format='%h %s%n%b%n---' "${last_version_commit}..HEAD"`.
   - Existing bullets are authoritative for **content** — preserve every recorded user-visible change, keep their order, don't merge them away. They are **not** authoritative for **form or truth**: any existing `[Unreleased]` bullet over the § 1 hard cap of 25 words gets split before you add yours, any bullet braiding more than one claim gets split, and any claim you can't pin to a `file:line` gets narrowed or cut. Then *add* what's missing, under the right sub-heading (`feat`→**Added**, `fix`→**Fixed**, user-visible `refactor`/`perf`/`style`→**Changed**, removals/breaking→**Removed**), in oldest→newest order within each sub-heading. Skip any effect already covered by an existing bullet.
   - Apply the canonical changelog style rubric in `.agent/workflows/package.md` § 1 (full rules + a before/after example live there). Load-bearing summary: **one bullet = one sentence = one user-visible effect, ≤ 25 words (a hard cap)**; the mechanism/why goes in the commit body, not here; skip file paths, type/function names, PR numbers, and CI/dep-bump churn.
   - If the change has no user-visible effect (pure refactor, internal-only, test/CI/docs), leave `CHANGELOG.md` untouched — there's simply nothing to add and it won't join the staged set.
7. Stage your authored set plus `CHANGELOG.md` (if you edited it) explicitly with `git add <file1> <file2> ...` — always name each file.
8. If dirty or untracked files exist outside your set, list them in your response under "**Not staged (not authored this session):**" so the user knows they were skipped on purpose.
9. Commit with the drafted message via HEREDOC. Stop after `git commit` — pushing is the user's call. Use plain `git commit -m`; leave `--no-verify`, `--amend`, and other rewrite flags out.
10. After committing, run `git status` and confirm the commit hash + landed files (including `CHANGELOG.md` when touched) in your response.
